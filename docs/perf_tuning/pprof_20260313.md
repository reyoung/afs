# 2026-03-13 Compose 性能与 pprof 记录

这份记录对应本地 Docker Compose 环境：

- `3` 个 `afs_layerstore_grpcd`
- `3` 个 `afslet`
- `1` 个 `afs_proxy`

## 测试目标

对比以下两件事：

1. AFS 在冷态和热态下的执行差异
2. 当前 Go 服务的 `pprof` 是否暴露出明显的用户态 CPU 热点

## 冷热态结论

冷态：

- AFS `reconcile wall time`: `46.220s`
- AFS `execute wall time`: `103.880s`
- Docker `execute wall time`: `43.653s`
- AFS 端到端：`150.110s`

热态：

- AFS `reconcile wall time`: `0.028s`
- AFS `execute wall time`: `53.323s`
- Docker `execute wall time`: `43.833s`
- AFS 端到端：`53.358s`

直接结论：

- 冷态时，AFS 相对 Docker 的 execute 差距约 `60.227s`
- 热态时，这个差距收敛到约 `9.490s`
- Warmup 的主要收益不在 `runc` 真正执行阶段，而在冷态首次 `reconcile + mount/materialize`

AFS 自身从冷态到热态的下降最明显的题目：

- `candies_2021`: `76.275s -> 23.396s`，下降 `52.879s`
- `beechtree_2023`: `73.323s -> 26.368s`，下降 `46.955s`
- `biscuits_2020`: `49.891s -> 4.604s`，下降 `45.287s`
- `aplusb_2023`: `40.875s -> 6.041s`，下降 `34.834s`

## 日志证据

冷态慢点主要集中在 `mount_start_to_ready`，而不是 `runc_wait`。

几个典型样本：

- `aplusb_2023`
  - 冷态 `mount_start_to_ready=34103ms`
  - 热态 `mount_start_to_ready=603ms`
  - `runc_wait` 从约 `3243ms` 降到约 `2051ms`
- `beechtree_2023`
  - 冷态 `mount_start_to_ready=47348ms`
  - 热态 `mount_start_to_ready=403ms`
  - `runc_wait` 基本不变，约 `4.6s`
- `biscuits_2020`
  - 冷态 `mount_start_to_ready=40923ms`
  - 热态 `mount_start_to_ready=401ms`
- `candies_2021`
  - 冷态 `mount_start_to_ready=47346ms`
  - 热态 `mount_start_to_ready=402ms`

这说明 warmup 消掉的主要是：

- layer reconcile
- 首次 mount ready
- 首次 layer materialize / 本地可见化

而不是容器内编译器本身的算力问题。

- 没看到 manifest 解析或下载路径上的明显 CPU 热点

## 当前判断

这批 profile 支持以下判断：

1. 当前主要慢点不是 Go 服务自己的用户态 CPU 热点
2. 冷态的大头开销已经可以明确归因到首次 `reconcile + mount ready`
3. 热态下 AFS 仍然比 Docker 慢约 `9~10s`，但这部分更像：
   - FUSE / `fuse-overlayfs` 路径
   - 内核态 / 用户态切换
   - 文件系统路径上的 steady-state 成本

这批 profile 不支持以下结论：

- 不能仅靠 Go pprof 证明 FUSE 路径没问题
- 不能仅靠 Go pprof 解释热态剩余 `9~10s` 的差距

## 下一步建议

如果继续定位热态剩余差距，优先级建议是：

1. 在热态基准下对 `afs_mount` / `fuse-overlayfs` / 容器内编译进程做 `perf record`
2. 对比 Docker overlay2 与 AFS FUSE 路径的 sys time
3. 再决定是否需要继续增强 `afslet` 的调度时间戳或 trace
